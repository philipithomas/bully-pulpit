# bully-pulpit

Personal website and blog for philipithomas.com. Next.js with MDX content. Hosted on Vercel.

## Development

```bash
nvm use 24
pnpm install
pnpm dev          # Start dev server with Turbopack
pnpm build        # Production build
pnpm test         # Run Vitest tests (unit + PGlite integration)
pnpm check        # Biome lint + format check
pnpm images:optimize  # Resize/compress public raster images to deployable web limits
pnpm content:check    # Validate generated artifacts are in sync (images, search index, related)
pnpm links:check      # Validate internal links/images in content MDX (offline)
pnpm canonical:check  # Scan built HTML: every indexable page has a canonical (runs as postbuild)
pnpm search:posts "query" # Search posts locally; use pnpm --silent search:posts --json "query" for agents
pnpm search:index     # Embed new/changed chunks → search-index.json + related-posts.json
pnpm workflow:check   # Validate generated Workflow handlers + queue triggers (runs as postbuild)
pnpm workflow:smoke   # POST a no-op Workflow smoke run to a deployed URL
pnpm db:generate  # Generate a Drizzle SQL migration from schema changes
pnpm db:migrate   # Apply migrations to DATABASE_URL (Neon)
pnpm db:studio    # Open Drizzle Studio against the DB
```

## Architecture

- **Content**: MDX files in `content/{contraption,workshop,postcard,tsundoku,pages}/`. New-post pipeline: add MDX (+ cover under `public/images/covers/` or post photos under `public/images/posts/`) → run `pnpm images:optimize public/images/path/to/image.jpg` for every new deployable raster image → `pnpm search:index` → commit the generated artifacts. The image optimizer skips files that already fit policy, so reruns are safe and do not keep recompressing the same image. Public raster images are normal Git files, not Git LFS files; keep camera originals outside this app repo and commit only web masters. `pnpm content:check` (offline, fast) validates all of it — every public JPG/PNG/WebP/AVIF is at most 5120px on the longest edge and 8MB on disk, image sources are readable real files rather than LFS pointers, the committed search index matches a recomputed merkle tree over the post corpus, every post has a related-posts entry — and runs in **pre-commit, CI, and the Vercel build command**, so a forgotten step fails the deploy instead of silently shipping degraded pages/search. Web images, social preview images, and newsletter email images rely on Vercel Image Optimization at render/fetch time rather than committed resized variants
- **Image check failures**: If `pnpm content:check` fails because a public image is over 5120px, over 8MB, or unreadable as an image, fix the named file before changing code: export RAW/camera-original photos to JPG/PNG/WebP/AVIF if needed, run `pnpm images:optimize public/images/path/to/image.jpg`, then rerun `pnpm content:check`. If the failure says the file is a Git LFS pointer, replace it with the real optimized image file; do not re-enable Git LFS for `public/images`
- **Rendering**: All public pages statically generated via `generateStaticParams`. The exception is the Printing Press admin (`/printing-press/*`), which renders dynamically: its layout awaits `requireAdmin()`, which reads the session cookie per request. **Canonical guard**: `pnpm canonical:check` (`scripts/check-canonical.ts`) scans the prerendered HTML in `.next/server/app/**/*.html` after every build and fails if any indexable page lacks exactly one `<link rel="canonical">` on the production host. It runs automatically as the `postbuild` npm hook, so it fires in pre-commit, CI, and the Vercel build with no extra wiring. The indexable/noindex partition is self-deriving from the HTML (a page that emits `<meta name="robots" content="noindex">` is exempt; the exemption is itself asserted), so the guard needs no per-route maintenance as content grows. Utility pages (`/account`, `/unsubscribe`) set `robots: { index: false }` in their metadata and are the intended exemptions
- **Auth**: Client-side overlay; subscribers/OTP/magic-link in Neon Postgres; session is a JWT (jose, HS256) in the httpOnly `bp_token` cookie (`bp_has_session` mirrors login state for the client)
- **Database**: Neon Postgres via Drizzle ORM (`src/lib/db/`). DB clients are lazy so most `next build` paths do not need `DATABASE_URL` — keep DB access in route handlers / server actions / workflows, never at module scope. The homepage subscriber count is the deliberate exception: it makes a best-effort `countActive()` read during static generation so the homepage can render a stale-but-fast subscriber count without a client-side flash, and fails open to no count when build-time DB access is unavailable. The `vercel.json` build command is `content:check && db:migrate:deploy && pnpm build`: the offline content check runs first (fails fast before any state change), then migrations, then build. Migrations are **guarded to `VERCEL_ENV=production`** — preview builds must not run migrations. `db:migrate` uses `DATABASE_URL_UNPOOLED` when present. Use additive (expand/contract) migrations so a mid-deploy schema/code gap stays backward-compatible
- **Email**: AWS SES (`@aws-sdk/client-sesv2`). Transactional sends (OTP, admin notification) are synchronous; the newsletter blast runs through a durable Vercel Workflow (`src/workflows/send-newsletter.ts`). Delivery is at-least-once (no SES idempotency key). Per-recipient rows snapshot the rendered HTML + plaintext (`text_content`) so the text/plain part carries the real body. Newsletter images point at first-party `/_next/image` URLs on the production host (`w=640` for covers/body images, `w=256` for related thumbnails) so Vercel resizes them dynamically for email clients; there are no committed `public/images/email` variants. A monthly cron (`/api/cron/subscriber-backup`, first of the month at 11:00 UTC) emails admins the full subscriber list as a CSV attachment via a hand-rolled raw MIME message (`src/lib/email/mime.ts`). Bounce/complaint capture is **polling only** (deliberately no SNS/webhook — no SES-side setup): a 15-minute cron (`/api/cron/suppression-sync`) mirrors the SES account-level suppression list into `email_suppressions` and is authoritative for its source (rows SES drops are deleted); the subscribers panel surfaces when and why an address is suppressed. Every `/api/cron/*` route requires `Authorization: Bearer $CRON_SECRET`; Vercel sends that header automatically to cron paths when `CRON_SECRET` is set
- **Workflow SDK upgrades**: keep `workflow` pinned to an exact version (no caret). Upgrade only in a PR with `pnpm add workflow@x.y.z -w --save-exact`, then run `pnpm test`, `pnpm check`, and `pnpm build`. `postbuild` runs `pnpm workflow:check`, which asserts that the SDK generated `/.well-known/workflow/v1/{flow,step,webhook}` plus the expected `queue/v2beta` trigger config and manifest entries for the newsletter and smoke workflows. After a local `vercel build --target preview`, run `CHECK_VERCEL_WORKFLOW_OUTPUT=1 pnpm workflow:check` to assert the final `.vercel/output/functions/.well-known/workflow/v1/{flow,step}.func/.vc-config.json` queue triggers. After Vercel builds the preview, run `WORKFLOW_SMOKE_BASE_URL=https://<preview-host> CRON_SECRET=<preview-secret> pnpm workflow:smoke`; this hits the CRON_SECRET-protected `POST /api/cron/workflow-smoke` route, which starts the no-op `src/workflows/workflow-smoke.ts` workflow and waits for it to complete. Do not validate a Workflow SDK bump by sending a newsletter. After merging and production deploy, run the same smoke command against `https://www.philipithomas.com` before publishing any newsletter. If the smoke response stays `pending`/`running` or times out, inspect with `pnpm exec workflow inspect run <run-id> --backend vercel --project bully-pulpit --team team_Axozqs0tBzA8ZbwDMpVUUhX2 --env production --json`, cancel it with `pnpm exec workflow cancel <run-id> --backend vercel --project bully-pulpit --team team_Axozqs0tBzA8ZbwDMpVUUhX2 --env production`, and roll back or pin the package back before any real send
- **Phone**: Twilio voice and SMS webhooks under `src/app/api/phone/` (`voice`, `voice-menu`, `recording-complete`, `recording-status`, `sms`), thin wrappers over `src/lib/phone/` — ported from junk-drawer's Rails phone backend. Auth on every webhook is a shared `?secret=` query param using `TWILIO_SECRET`, embedded in the webhook URLs configured in the Twilio console, compared in constant time and failing closed when unset. The voice route speaks an AI-generated greeting (AI Gateway `anthropic/claude-haiku-4.5`, prompted with NYC time + wttr.in weather; any failure falls back to a static greeting so the call always proceeds) via Twilio `<Say>`, then gathers one digit: 1/no input records a voicemail, 2 subscribes the caller ID to all SMS newsletters. A completed recording starts the durable voicemail workflow (`src/workflows/process-voicemail.ts`): download the MP3 from Twilio → transcribe via OpenAI (`OPENAI_API_KEY` — the AI Gateway does not route transcription models) → email the transcription with the MP3 attached (the raw-MIME attachment path in `src/lib/email/mime.ts`). Recordings under 1000 bytes and empty transcriptions are skipped as butt dials. Phone notifications (missed call, voicemail, inbound SMS, new SMS opt-ins) go to the `ADMIN_EMAILS` allowlist (every address, via `phoneNotificationRecipients()` in `src/lib/phone/config.ts`); if the allowlist is ever blanked it falls back to the static philip@contraption.co so a phone notification is never dropped. **Text messaging**: inbound SMS is persisted to a `text_messages` table (`src/lib/db/queries/text-messages.ts`, deduplicated by `twilio_sid`) before the email notification fires; inbound `SUBSCRIBE`/`START`/`JOIN` commands upsert `sms_subscribers` and opt the number into every newsletter, while `STOP`/`STOPALL`/`UNSUBSCRIBE`/`CANCEL`/`END`/`QUIT`/`REVOKE`/`OPTOUT` clear local SMS eligibility. `SUBSCRIBE` returns a branded compliance confirmation that identifies recurring new-post texts, variable frequency, possible message and data rates, HELP, and STOP. When Twilio has not already handled the keyword, `HELP` returns `mail@philipithomas.com` with the frequency, rate, and STOP details. Voice-menu signups send the same subscription confirmation SMS when Twilio accepts it and otherwise speak the STOP instruction. A number that transitions from inactive to active through either the text or voice-menu path also receives one Bell onboarding MMS; repeating `SUBSCRIBE` while active does not resend it, but reactivation after an unsubscribe does. The MMS attaches the stable `/bell.vcf` permalink, which returns a vCard 3.0 contact named Bell with the sending number, the Philip I. Thomas organization and website, and an embedded Bell image. iPhone opens the attachment in its native contact preview, but the person must manually create or update the contact; the app cannot save it silently. Twilio Advanced Opt-Out webhooks never duplicate Twilio's START, STOP, or HELP response; START and STOP still sync local state, and a newly active START subscriber receives only the distinct Bell onboarding MMS from the app. The configured Twilio response remains user-visible for every classified keyword, so its START and HELP copy must stay aligned with the app's program description, support address, frequency, rate, HELP, and STOP disclosures. Every other inbound text with a `MessageSid` starts `src/workflows/reply-to-sms.ts` under a reclaimable webhook lease: Bell receives the 12 most recent delivered messages through that inbound row (including automated newsletter notices), generates with the same model and read-only tools as web Bell, and sends one REST reply prefixed `[Bell AI]`. The formatter strips Markdown, normalizes smart punctuation, and caps the complete body at 300 GSM-7 units or 132 UCS-2 code units so the reply stays within two toll-free segments; a fixed reply is used if generation exhausts its retries. Workflow and Twilio delivery remain at-least-once because Twilio has no idempotency key; `reply_to_message_id` keeps the local history idempotent, but a crash after provider acceptance can still duplicate the external text. Unsubscribe also marks pending unsent `sms_sends` rows for that number as skipped, and retry does not clear that skipped marker, so an old queued post is not sent after someone opts out and later resubscribes. First active SMS opt-ins from text or voice-menu email admins with the source path, Twilio metadata when present (`FromCity`, `FromState`, `CallerName`, `MessageSid`, `CallSid`), and an area-code hint for common NANP numbers. Newsletter publishes enqueue one `sms_sends` row per eligible number and send from `PHONE_NUMBER` after the email workflow pass, with outbound texts also persisted to `text_messages` for the Printing press Phone panel. The send preview page can also send a one-off test SMS to `OWNER_PHONE_NUMBER` (private E.164 env var) without adding that personal number to source control. SMS post links include one attribution parameter (`utm_source=sms`). The public subscribe prompt uses `PHONE_NUMBER` on subscribe screens. The Phone panel (`/printing-press/phone`) lists conversations grouped by the other party's number (newest first) and renders inbound/outbound threads, all admin-guarded. Outbound SMS and click-to-call go through `src/lib/phone/twilio.ts` (form-encoded REST against `api.twilio.com`, HTTP Basic auth with `TWILIO_SID`/`TWILIO_SECRET` — the account SID and auth token, in `accountSid:authToken` order — no-op mock off production). Click-to-call rings `OWNER_PHONE_NUMBER` first, then on answer Twilio fetches the secret-validated `POST /api/phone/connect?secret=...` callback which returns XML-escaped `<Dial callerId>` to the destination; `callerId` comes from `PHONE_NUMBER`. All destination numbers are E.164-validated; SMS bodies are length-bounded. Cutover: set the env vars in Vercel, point the Twilio number's voice webhook at `POST /api/phone/voice?secret=...` and the messaging webhook at `POST /api/phone/sms?secret=...#rt=5000&rc=2&rp=5xx,ct,rt` (the retry policy is required for reclaiming failed workflow enqueues), then retire junk-drawer's `/twilio/*` routes
- **SMS signup UI flag**: Vercel flag `sms-signup-ui` gates only the public signup affordances: `SmsSubscribePrompt` on web subscribe surfaces and the phone voice route's "press 2" menu option. It defaults off in code, production should stay off until launch, and preview/development can be enabled in Vercel for testing. Backend SMS webhook handling remains active so `STOP`, Twilio redeliveries, and manual `SUBSCRIBE` texts continue to work safely.
- **Admin (Printing Press)**: the admin panel lives at `/printing-press` (`/admin` 307-redirects there), gated by the `ADMIN_EMAILS` allowlist (`src/lib/auth/admin.ts`) on top of a normal signed-in session — page layout guard (`requireAdmin()`) + per-route `guardAdmin()` on every `app/api/printing-press/*` handler. The shell (`src/components/printing-press/`) is a single centered column: serif "Printing press" masthead over a row of plain text nav links (no sidenav, Sheet drawer, or icons; the public site header above links home). Sections: Overview, Posts (send UI), Subscribers (search by email, gravatar avatars, copy, hard-delete, suppression clearing — removes the address from the SES account-level suppression list and the local `email_suppressions` row, SES first so a failed clear leaves visible state, CSV export/import — export columns `email,name,postcard,contraption,workshop,tsundoku,confirmed,source,created_at`, import upserts by email and also accepts optional `source`; flag columns absent from the CSV only set defaults on new rows, so a bare email list can't re-subscribe opt-outs; `source` is backfill-only on existing rows (fills NULL or the `csv_import` placeholder, never overwrites a real captured referrer, so an attribution re-sync is idempotent); export neutralizes spreadsheet formula injection in `email`/`name`/`source`), and Phone (SMS conversation view + outbound SMS + click-to-call, see Phone above). Gravatar avatars knowingly send md5(subscriber email) to Automattic — accepted tradeoff for an admin-only list. The dropdown's "Printing Press" link shows only when `/api/auth/me` reports `isAdmin`
- **Search**: fully local, no external search service. `src/lib/search/` holds the pipeline: `corpus.ts` deterministically chunks every published post and every content page in `content/pages/` (title chunk, ~1200-char body chunks broken at headings, cover-alt chunk) and extracts every local public cover/body image from posts as a searchable image asset with title, newsletter, role, nearest heading, alt text, and optional image-file digest for offline checks. Content pages use `newsletter: "page"` and return `type: "page"` from search. `pnpm search:index` embeds text chunks and image assets with Google Gemini Embedding 2 (`google/gemini-embedding-2`, 768 dims + L2-normalized) and commits them to `src/generated/search-index.json` (hashes + base64 vectors only, no raw text) plus `src/generated/related-posts.json` (post-only centroid cosine similarity across text and image vectors, score = 1 - cosine). Text embedding uses the AI SDK gateway provider; multimodal image+description embedding uses the Vercel AI Gateway OpenAI-compatible `/v1/embeddings` endpoint with `AI_GATEWAY_API_KEY` because the AI SDK embedding adapter is text-only. A merkle tree over chunk text and image metadata/file hashes (`merkle.ts`) lets `content:check` verify the committed index offline and name the stale slugs; the script reuses vectors for unchanged hashes so a routine run embeds only new or changed text/images. Staleness is guarded by pre-commit, CI, and the Vercel build command via `content:check`: a content or image change that did not regenerate the index fails with `run pnpm search:index`, no regen, no embedding call. `hybridSearchPosts` is the shared search path for typeahead and Bell: MiniSearch BM25 (`lexical.ts`, title boosted 6x, image text indexed separately) is fused with brute-force cosine over committed vectors via weighted reciprocal rank fusion (70% BM25, 30% vector) and degrades to BM25 if query embedding fails or exceeds the 800 ms timeout. `/api/search` defaults to site content results and accepts `scope=images` for image-only results; `/photography` exposes that image scope as an inline photo search, and the global search dialog sends Vercel Analytics properties for resolved mode, result count, server duration, and client duration. `pnpm search:posts "query"` is the agent-friendly CLI: it defaults to credential-free BM25 over the same corpus, supports `--json` for machine-readable output (`pnpm --silent search:posts --json "query"`), and accepts `--hybrid` when AI Gateway auth is available and semantic query matching matters
- **Chat (Bell)**: site chat sidebar (`src/components/chat/`, `app/api/chat`) streams via AI SDK + AI Gateway (`anthropic/claude-haiku-4.5` on gateway credits, no BYOK) with tools `searchPosts` (local hybrid search over posts, content pages, and `scope: "images"` for photo/image questions, see Search above), `fetchPost`, and `fetchPage` (path → readable text via `src/lib/chat/page-content.ts`: hand-written homepage summary, newsletter indexes with the five latest posts, content pages as plaintext, a not-found line otherwise, every response bounded), capped at 7 steps with the last step forced to prose (`activeTools: []`). `searchPosts` and `fetchPost` return image metadata, and the chat renderer shows compact linked thumbnails from tool output so Bell can answer questions about photos such as coffee images with visible evidence. The client sends the current pathname and `document.title` with each request; the route validates both fail-open (conservative path pattern, 200-char caps, invalid values silently dropped, never a chat failure) and appends a current-page line to the system prompt — for posts and content pages the page text itself is injected (`src/lib/chat/page-context.ts`), elsewhere the prompt directs the model to `fetchPage`. Latency: gateway routing is pinned to Anthropic direct (`order: ['anthropic']`, `sort: 'ttft'` — Bedrock/Vertex serve the same model with slower time-to-first-token), and the pre-stream checks (Firewall rate limit, BotID, body parse) run concurrently since both checks read only headers. Anthropic prompt caching is deliberately not wired: Haiku 4.5 needs a 4096-token minimum cacheable prefix and Bell's system prompt tops out around 2k tokens, so a cache marker would silently never engage. The client renders only text and tool parts, and scrubs leading tool-call JSON from text (`src/lib/chat/scrub-leaked-tool-json.ts`) — gpt-oss-120b used to leak unbound call JSON into replies; keep that guard even though the model changed
- **Styling**: Tailwind v4 CSS-first config in `src/styles/globals.css`

## Conventions

- Path aliases: `@/*` maps to `./src/*` (relative imports banned by biome)
- Biome for linting/formatting (not ESLint/Prettier)
- Single quotes, no semicolons (biome config)
- Integration tests are `*.integration.test.ts`, colocated, and run real SQL against an in-memory PGlite Postgres (no Docker/network) in the same `pnpm test` run. Harness at `src/test/integration/`: swap the Neon client with `vi.mock('@/lib/db/client', () => import('@/test/integration/db'))` (applies the real migrations, fresh DB per file, `resetDb()` per test), `session.ts` replaces `next/headers` for real-JWT sessions, `mocks.ts` has SES/BotID factories. Mock only external I/O — never `@/lib/db/queries/*`
- Fonts loaded from fonts.philipithomas.com CDN
- GitHub workflows (`.github/workflows/`): `check.yml` is CI (every push/PR to main: Biome, tsc, tests, `content:check`, `links:check`, build); `postcard-draft.yml` opens a draft postcard PR monthly. External links are not checked automatically because historical references, paywalls, bot blocking, and rate limits make automated reports noisy. There are no Claude automation workflows
- Repo-local Codex skills live in `.codex/skills/`. `copyedit` and `critique` replace the old Claude skills for MDX editorial workflows
- Pre-commit hook runs lint-staged, `pnpm content:check`, and `pnpm build` — no need to build manually before pushing

## Prose style guide

ALL user-visible text follows the style guide in `content/pages/colophon.mdx` — blog posts and page copy, but also UI strings, toasts, empty states, email templates, and product naming. The style guide now explicitly says to embrace being weird: keep personality, quirks, texture, and deliberate rule breaks when they make the writing more itself. The rules that bite most in UI code: sentence case for titles and names ("Printing press", not "Printing Press"), no contractions, no em dashes (use periods or colons), ISO-8601 dates, active voice. The admin panel is deliberately not metrics-driven — prose summaries over stat cards; never add open/click tracking.

## Newsletters

- **Contraption** (`/contraption/`): Essays and launches. Forest (#2B4A3E) accent.
- **Workshop** (`/workshop/`): Work in progress notes. Walnut (#6B4D3A) accent.
- **Postcard** (`/postcard/`): Monthly updates. Indigo (#2C3E6B) accent.
- **Tsundoku** (`/tsundoku/`): Japan photo posts. Rising sun (#E20612) accent and a light neutral `#f4f4f2` page and email background.

New public signups are subscribed to every newsletter, including Tsundoku, regardless of which signup form they used. Existing subscribers who submit a newsletter-specific form are opted into only the requested newsletter(s); if that adds Tsundoku, the admin notification email is sent. Signed-in users who are not subscribed to the current newsletter see the generic post/page subscribe CTA, which patches only that newsletter preference.

Sending: an admin opens `/printing-press` → Posts, picks a post, previews it, sends a test to themselves, then sends to all confirmed email and SMS subscribers of that newsletter. The send is a durable Vercel Workflow that enqueues per-recipient `email_sends` rows and delivers them in paced, retryable batches via SES (transient errors back off via `RetryableError`); after the email pass it enqueues `sms_sends` rows and texts eligible SMS subscribers through Twilio at 1 message per second for the single long-code sender. Twilio 429/5xx/network errors back off with `RetryableError`; recipient-specific 4xx errors become row-level `send_error`s. Eligibility excludes anyone already sent or pending; the Send path heals previously-errored rows in place (`resetFailedBySlug`, `resetFailedSmsBySlug`) so a re-send reuses them instead of duplicating. Both the Send and Retry routes gate on real workflow run liveness, not on pending send rows: each `start()` persists its runId in `send_runs` (keyed by slug) and the guard (`isSendRunActive`, `src/lib/email/send-guard.ts`) asks the runtime `getRun(runId).status` — only `pending`/`running` returns 409. Pending rows OUTLIVE a dead run (throttle-exhausted, cancelled, crashed), so inferring "in flight" from them would deadlock the Retry resume path forever; run status distinguishes a live run from a stalled send that Retry is meant to take over. UI primitives live in `src/components/ui/` (Base UI where a headless primitive is needed, plus cva) on the shadcn token contract: `globals.css` defines the shadcn CSS variables (`--background`, `--primary`, `--destructive`, …) mapped to the warm palette. Rules: never run `shadcn init` (it rewrites globals.css and breaks the @font-face setup); install deps with `-w` (this repo is a pnpm workspace root); when updating a shadcn primitive, use the Base UI source/docs and port it manually while preserving this repo's `new-york` component config and warm tokens; every vendored component gets an editorial pass — strip shadows and `.dark` blocks, remove rounding from interactive controls (buttons and inputs are square; `rounded-full` only for avatars and tag-style pills), sentence-case strings per the colophon, and swap stock palette utilities for warm tokens (`destructive`/site `red` for danger, `forest`/`brass` tints for status — the warm gray scale is spelled `gray-050`, not `gray-50`). Focus follows a two-tier contract in globals.css: the global `:focus-visible` outline is the keyboard focus indicator for everything except text entry, and text-entry controls (inputs other than checkbox/radio, textareas, selects) are excluded at the selector level — their single indicator is a global rule darkening the border to gray-900 on focus. Do not add per-component focus rings or `focus:border-*` utilities. The public `.btn`/`.btn-primary`/`.btn-indigo` classes in globals.css are the marketing-CTA tier (signature arrow-reveal hover) and stay hand-built outside the token contract.
